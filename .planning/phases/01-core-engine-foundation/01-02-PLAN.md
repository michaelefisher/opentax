Create `engine/core/runtime/planner.ts` exporting `buildExecutionPlan`.

The planner takes two inputs:
1. `registry: NodeRegistry` - all registered nodes
2. `inputs: Record<string, unknown>` - the raw inputs.json content (what the start node receives)

It returns an ordered array of `ExecutionStep`:
```typescript
export type ExecutionStep = {
  readonly id: string;       // instance ID (e.g., "start", "w2_01", "w2_02", "line_01z_wages")
  readonly nodeType: string; // which TaxNode to use from registry
};
```

Algorithm:
1. Start with the "start" node. Run its compute() with the provided inputs to get its outputs.
2. For each output from start, create an instance. If a nodeType appears multiple times, suffix with `_01`, `_02`, etc. If it appears once, use the nodeType as the ID.
3. For each instance, look up its node in the registry and record its `outputNodeTypes` as downstream dependencies.
4. Downstream singleton nodes (like `line_01z_wages`) get the nodeType as their ID.
5. Build an adjacency list from instance IDs and topologically sort (Kahn's algorithm - BFS with in-degree tracking).
6. Return the sorted `ExecutionStep[]`.

Important: The planner must actually run start.compute() to know which instances to create (since start decides based on what forms are present in inputs). For non-start nodes at planning time, we use `outputNodeTypes` metadata only (not compute) to build the rest of the graph.

Create `engine/core/runtime/executor.ts` exporting `execute`.

The execute function:
```typescript
export type ExecuteResult = {
  readonly pending: Readonly<Record<string, Record<string, unknown>>>;
};

export function execute(
  plan: readonly ExecutionStep[],
  registry: NodeRegistry,
  inputs: Record<string, unknown>,
): ExecuteResult
```

Algorithm:
1. Initialize `pending: Record<string, Record<string, unknown>> = {}`.
2. Seed the start node: `pending["start"] = inputs`.
3. For each step in plan order:
   a. Look up node from `registry[step.nodeType]`.
   b. Get input from `pending[step.id] ?? {}`.
   c. Run `node.inputSchema.safeParse(input)`.
   d. If parse fails, skip silently (continue). Per EXEC-03.
   e. If parse succeeds, call `node.compute(parsed.data)`.
   f. For each output in result.outputs: call `mergePending(pending, targetId, output.input)`.

The `mergePending` function (internal helper):
- For each key in `output.input`:
  - If target field is an array AND incoming value is not an array: append to array.
  - If target field is an array AND incoming value is an array: concat arrays.
  - If target field does not exist: set directly.
  - If incoming value is an array and target is not: convert target to array format if needed, else set.
  - Scalar fields: set (last writer wins).

Target ID resolution: When a node outputs to a nodeType, the target ID is the nodeType itself for singleton nodes. The planner has already resolved instance IDs, so the executor uses the nodeType from NodeOutput directly as the pending key for downstream singletons.

Per EXEC-02: execute in topo order, accumulate via pending dict.
